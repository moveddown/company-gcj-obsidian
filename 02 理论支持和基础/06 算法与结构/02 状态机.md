状态机事件处理有两种常见写法：

1. 单事件槽版本

结构体里只有一个 event 字段。
外部调用 PostEvent(event) 时，直接把事件写入 ctrl->event。
状态机 Run() 时读取 ctrl->event，处理完成后清空为 EVENT_NONE。

优点：简单。
缺点：如果多个事件连续到来，后来的事件会覆盖前面的事件。

2. 事件队列版本

结构体里包含一个事件队列。
外部调用 PostEvent(event) 时，把事件 push 到队列尾部。
状态机 Run() 时，从队列头部 pop 一个事件处理。

优点：事件不会轻易丢失，适合多事件来源。
缺点：代码复杂一些，需要考虑队列满、中断并发、事件优先级。


==后续可以扩展为==
==普通 FIFO 队列、优先级事件队列、故障事件直接抢占==


优先级事件队列：

普通事件队列按 FIFO 处理事件，即先来先处理。
优先级事件队列会根据事件重要程度决定处理顺序，高优先级事件可以先于低优先级事件被处理。

常见实现方式：
1. 多个 FIFO 队列：每个优先级一个队列，取事件时从高优先级队列开始取。
2. 单数组排序：插入事件时按优先级排序。
3. 紧急事件抢占：普通事件走队列，关键事件单独保存或直接处理。

嵌入式工程中推荐先掌握多个 FIFO 队列方式：
- 每个优先级内部仍然保持先来先处理；
- 不同优先级之间，高优先级先处理；
- 实现简单，逻辑清楚，适合裸机 C 工程。


# 状态机事件处理这几种写法的==注意点总结==
下面是状态机事件处理这几种写法的**注意点总结**，适合直接记笔记。

# 1. 单事件槽版本注意点

单事件槽结构通常是：

```c
typedef struct
{
    State_t state;
    Event_t event;
} Ctrl_t;
```

特点是只有一个 `event` 字段。

## 注意点

### 1. 事件可能被覆盖

```c
ctrl->event = EVENT_A;
ctrl->event = EVENT_B;
```

最终只剩 `EVENT_B`，`EVENT_A` 丢失。

所以单事件槽适合：

```text
事件少
事件不密集
主循环处理很快
事件丢失影响不大
```

不适合：

```text
按键、定时器、串口、ADC、故障检测等多个事件源同时存在
```

---

### 2. 处理完事件后要清空

状态机处理完当前事件后，要写：

```c
ctrl->last_event = ctrl->event;
ctrl->event = EVENT_NONE;
```

否则同一个事件会在下一轮主循环中被重复处理。

---

### 3. 中断里不要直接跑复杂状态机

中断里建议只做：

```c
PostEvent(EVENT_TIMEOUT);
```

不要在中断里写大量状态机逻辑。

原因：

```text
中断执行时间要短
状态机逻辑可能较复杂
避免中断和主循环共享资源混乱
```

---

# 2. 普通事件队列版本注意点

普通事件队列是 FIFO：

```text
先投递的事件，先处理
```

## 注意点

### 1. 队列可能满

事件队列一般是固定长度数组：

```c
#define EVENT_QUEUE_SIZE 8
```

如果队列满了，再投递事件会失败。

所以 `PostEvent()` 最好返回结果：

```c
uint8_t PostEvent(Event_t event);
```

用于判断是否投递成功。

---

### 2. 重要事件不能随便丢

比如：

```text
过流
堵转
急停
过温
通信断开
```

这些事件如果队列满了还丢掉，可能有安全风险。

可以考虑：

```text
重要事件单独处理
重要事件使用优先级队列
队列满时覆盖低优先级事件
直接进入安全状态
```

---

### 3. 队列读写需要考虑并发

如果：

```text
中断里 Push 事件
主循环里 Pop 事件
```

那么队列就是共享资源。

需要保护临界区：

```c
__disable_irq();
EventQueue_Push(...);
__enable_irq();
```

或者采用更安全的设计：

```text
中断只置标志
主循环检测标志后再 PostEvent
```

---

### 4. 每次状态机可以处理一个事件，也可以处理多个事件

常见写法是每轮处理一个事件：

```c
if (EventQueue_Pop(&event))
{
    StateMachine_HandleEvent(event);
}
```

优点：

```text
单次运行时间可控
适合主循环
```

也可以一次处理完队列：

```c
while (EventQueue_Pop(&event))
{
    StateMachine_HandleEvent(event);
}
```

优点：

```text
事件响应更快
```

缺点：

```text
如果事件很多，可能阻塞主循环其他任务
```

---

# 3. 优先级事件队列注意点

优先级事件队列的目标是：

```text
重要事件先处理
```

常见优先级：

```text
CRITICAL：过流、堵转、急停
HIGH：停止键
NORMAL：启动键、复位键、普通超时
LOW：周期刷新、普通检测
```

---

## 注意点

### 1. 高优先级事件可能饿死低优先级事件

如果高优先级事件持续进入：

```text
OVER_CURRENT
OVER_CURRENT
OVER_CURRENT
```

低优先级事件可能一直得不到处理。

这叫：

```text
饥饿 starvation
```

解决方式：

```text
限制每轮高优先级事件处理数量
低优先级等待太久后提升优先级
过滤重复高优先级事件
```

---

### 2. 重复事件可能需要合并

有些事件重复投递没有意义：

```text
连续 10 个 STOP_KEY
连续 10 个 OVER_CURRENT
连续 10 个 TEMP_TIMEOUT
```

可以考虑：

```text
同类事件已在队列中，就不重复加入
状态已经进入 FAULT 后，忽略后续故障事件
周期事件只保留最新一次
```

否则队列可能被无意义事件占满。

---

### 3. 极端安全事件可以不排队

比如：

```text
急停
严重过流
硬件保护触发
```

这些事件可以直接置安全标志：

```c
volatile uint8_t emergency_stop_flag;
```

状态机运行前优先检查：

```c
if (emergency_stop_flag)
{
    EnterStopState();
    return;
}
```

原因：

```text
安全事件不应该排队等待
```

---

### 4. 每个优先级内部仍然建议 FIFO

推荐结构：

```text
CRITICAL 队列：FIFO
HIGH 队列：FIFO
NORMAL 队列：FIFO
LOW 队列：FIFO
```

处理顺序：

```text
先取 CRITICAL
再取 HIGH
再取 NORMAL
最后取 LOW
```

这样可以同时保证：

```text
不同优先级之间：重要事件先处理
同一优先级内部：先来先处理
```

---

# 4. 状态机动作设计注意点

状态机动作一般分三类。

## 1. 转移动作

动作属于某一条转移路径：

```text
当前状态 + 当前事件 → 动作 → 下一个状态
```

例子：

```c
if (event == EVENT_OVER_CURRENT)
{
    fault_code = FAULT_OVER_CURRENT;
    EnterFaultState();
}
```

适合：

```text
动作依赖事件来源
动作依赖故障类型
动作依赖从哪个状态来
```

---

## 2. 进入状态动作

动作属于目标状态：

```text
只要进入这个状态，就执行同样动作
```

例子：

```c
static void EnterStopState(void)
{
    Motor_Stop();
    Timer_Stop();
    state = STATE_STOP;
}
```

适合：

```text
进入停止状态一定要关电机
进入故障状态一定要关输出、开报警
进入自动状态一定要启动自动检测定时器
```

---

## 3. 状态内动作

动作在某个状态下周期执行或由周期事件触发。

例子：

```c
case STATE_AUTO:
    if (event == EVENT_1S_TIMEOUT)
    {
        CheckTemperature();
    }
    break;
```

适合：

```text
周期检测
通信接收
超时判断
自动控制
```

---

# 5. 定时器和资源管理注意点

## 1. 谁开启，谁关闭

如果某个状态独占一个资源：

```text
进入 AUTO → 开启温度检测定时器
离开 AUTO → 关闭温度检测定时器
```

这样逻辑最清楚。

---

## 2. 安全状态可以兜底关闭资源

比如停止状态、故障状态：

```c
static void EnterStopState(void)
{
    Motor_Stop();
    Timer_StopAll();
}
```

即使某些定时器理论上没开，关闭一次也可以作为安全兜底。

但要注意可读性，不要在每个普通 case 里到处写无关的 `Timer_Stop()`。

---

## 3. 不要在状态 case 顶层执行不该每次执行的动作

错误示例：

```c
case STATE_AUTO:
{
    CheckTemperature();
    ...
}
```

这样只要进入 `STATE_AUTO` 的 case，无论什么事件都会检测温度。

更好：

```c
case STATE_AUTO:
{
    if (event == EVENT_TEMP_TIMEOUT)
    {
        CheckTemperature();
    }
}
```

动作应该绑定到具体事件，除非它确实是状态持续动作。

---

# 6. 结构体封装注意点

状态机上下文结构体一般包含：

```c
typedef struct
{
    State_t state;
    State_t last_state;

    Event_t current_event;
    Event_t last_event;

    Fault_t fault_code;

    uint8_t flag;
    uint32_t timer_count;

    EventQueue_t event_queue;
} Ctrl_t;
```

## 适合放进结构体的变量

```text
当前状态
上一次状态
当前事件
上一次事件
故障码
运行标志
计数器
状态相关缓存
状态机专用配置
事件队列
```

## 不适合放进结构体的变量

```text
只在一个函数里临时使用的变量
完全属于其他模块的数据
和该状态机无关的全局数据
```

---

# 7. 状态机写法总原则

可以记这几句：

```text
状态描述“现在处于什么阶段”。
事件描述“发生了什么触发条件”。
动作描述“系统要做什么”。
上下文结构体描述“状态机需要记住什么”。
事件队列解决“多个事件如何保存”。
优先级队列解决“重要事件如何先处理”。
```

工程写法通常是：

```text
事件源 → PostEvent → 事件队列 → StateMachine_Run → switch(state) → 判断 event → 执行动作 → 切换 state
```
